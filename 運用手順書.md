## 運用手順書（Cafe Platform）

### 1. 前提（想定環境）
本運用手順書は次の構成を前提とする。

- `cafe_api`: FastAPI（Uvicorn）
- `cafe_web`: Next.js（一般ユーザー）
- `cafe_admin`: Next.js（運営管理）
- DB: PostgreSQL + PostGIS
- バックグラウンド処理:
  - アナウンス公開ワーカー（announcement_worker）
  - ランキング更新ワーカー（ranking_worker）
  - クリーンアップワーカー（cleanup_worker）
  - メールワーカー（email_worker: arq）

---

### 2. 開発環境セットアップ（ローカル）
#### 2.1 `cafe_api` のセットアップ
1. Poetry環境を用意
2. `cafe_api/.env` を作成（必要値を設定）
3. DB作成後にマイグレーションを適用

`cafe_api` の既存ドキュメントに従い実行例:
- `alembic upgrade head`

#### 2.2 起動
開発サーバー例:
- `poetry run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000`

注意:
- `app/main.py` の lifespan で announcement worker は自動開始される（APIサーバー内のバックグラウンドタスク）。

#### 2.3 `cafe_web / cafe_admin` の起動
各ディレクトリで `pnpm install` → `pnpm dev` を実行する。

API base URL は `cafe_web/src/lib/api-client.ts` / `NEXT_PUBLIC_API_URL` を用いて設定する。

---

### 3. マイグレーション運用（DB変更）
#### 3.1 変更の反映手順
1. SQLAlchemyモデルを更新
2. `alembic revision --autogenerate -m "description"`
3. `alembic upgrade head`

#### 3.2 ロールバック
- `alembic downgrade -1`

---

### 4. 本番デプロイ運用（Docker想定）
本リポジトリには各サービスの Dockerfile と `cafe_api/entrypoint.sh` がある。

#### 4.1 `cafe_api` コンテナ起動
`cafe_api` Dockerfile は `SERVICE` 環境変数で起動モードを切り替える。

- `SERVICE=api`: FastAPIのみ
- `SERVICE=email-worker`: メールワーカーのみ
- `SERVICE=both`: API + メールワーカー（ただしランキング/クリーンアップは未自動のため別プロセスで起動するのが推奨）

推奨:
- APIコンテナとは別に `ranking_worker` / `cleanup_worker` を動かすコンテナ/タスクを用意する

#### 4.2 `cafe_web` / `cafe_admin` コンテナ起動
各Next.jsの Dockerfile は `nextjs` ユーザーで `node server.js` を起動する。

---

### 5. バックグラウンドワーカー運用
#### 5.1 announcement_worker（スケジュール投稿）
挙動:
- 1分間隔で `scheduled_at` 到達の投稿を `published_at` へ反映

運用手順:
- APIサーバー内で常時起動される（`app/main.py`）ため、基本的には別起動不要

#### 5.2 ranking_worker（ランキング）
挙動:
- 24時間間隔で `Store.is_active=true` を対象にスコア更新
- スコア: `favorite_count*0.4 + route_click_count*0.3 + view_count*0.3`

運用手順:
- 通常は `ranking_worker` を別プロセスとして常時起動する
- 更新が遅れる場合は、スケジュール間隔とコンテナ稼働状態を確認する

#### 5.3 cleanup_worker（期限切れ/古いログの削除）
挙動:
- Refresh token 期限切れ削除
- `user_events` / `store_events` の古いイベント削除（デフォルト90日保持）

運用手順:
- 24時間間隔で実行
- DBサイズが増大している場合は `days` を見直す前に監視・影響評価を行う

#### 5.4 email_worker（メール送信キュー）
挙動:
- `arq` を使用してメール関連の非同期処理を実行

運用手順:
- `SERVICE=email-worker` もしくは別ワーカープロセスとして常時起動

---

### 6. 設定管理（環境変数）
`cafe_api/app/core/config.py` の `Settings` に従い環境変数を設定する。

必須になり得る設定:
- `DATABASE_URL`（DB接続）
- `SECRET_KEY`（JWT署名）
- `ALGORITHM`（デフォルトHS256）
- `STRIPE_SECRET_KEY` / `STRIPE_PUBLIC_KEY`（決済有効化時）
- `SMTP_*`（メール有効化時）
- `GOOGLE_CLIENT_ID / SECRET` 等（OAuth有効化時）

注意:
- Dockerfile上ではSecretsをビルド時に埋め込まず、実行時環境変数から注入する前提

---

### 7. 運用チェックリスト
#### 7.1 デプロイ直後
- `/health` が `healthy` になること
- `stores/top/ranking` の更新が反映されること（ランキングバッチの反映は最大24h）
- 予約投稿が公開されること（最大1分）

#### 7.2 定期
- DBのテーブルサイズ/インデックス使用状況（特にイベントログ）
- refresh_tokens の残数（cleanupが機能しているか）

---

### 8. 障害時の切り分け（最小手順）
1. `cafe_api` の `/health` とログ（LoggingMiddlewareの Request/Response）を確認
2. `HTTP 500` が増えている場合、該当エンドポイントのdetailを追跡
3. バッチ系の場合:
   - ranking: `ranking_worker` のプロセス起動有無
   - cleanup: `cleanup_worker` の実行周期と実行ログ
4. Webhook系（Stripe）:
   - 現状 `POST /payments/webhook` はTODOのため、本番では実装状況と署名検証可否を確認
