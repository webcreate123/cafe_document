## システム要件定義書（Cafe Platform）

作成日: 2026-03-25

### 1. 背景・目的（要約）
国内で増加するカフェにおいて、(1) 混雑状況や在庫/セールなどの鮮度情報の共有不足、(2) 店舗側の適切な運用・人員確保、(3) ユーザー側のミスマッチ（閉店・SoldOut・入店待ち等）による機会損失が課題となっている。

本システムは「カフェ」と「来店を希望するユーザー」に対して、混雑/空席状況・セール・求人などを軸に相互価値を生むプラットフォームを提供する。

### 2. 対象範囲（スコープ）
対象システムは以下の3コンポーネントから構成する。

- `cafe_web`: 一般ユーザー向けWeb（Next.js）
- `cafe_admin`: 運営/管理者向けWeb（Next.js）
- `cafe_api`: バックエンドAPI（FastAPI / OpenAPI生成）

### 3. 利用者（ロール）と権限
`cafe_api` の定義に基づき、主に以下のロールを扱う。

- 一般ユーザー: `USER`
- 店舗オーナー: `STORE_OWNER`
- 店舗スタッフ: `STORE_STAFF`
- 運営管理者: `ADMIN`

RBAC（ロールベースアクセス制御）により、`/admin` などの運営API・管理画面へのアクセスを制限する。

### 4. 前提・制約
#### 4.1 技術前提
- バックエンド: FastAPI（OpenAPIを自動生成）
- フロントエンド連携: OpenAPIからTypeScript型/クライアント自動生成（`openapi-typescript-codegen`）
- DB: PostgreSQL + PostGIS（店舗の位置情報検索・半径検索のため）
- 認証: JWT（Access/Refresh）

#### 4.2 外部IF要件
本システムは、以下外部サービス連携を前提とする。

- 決済: Stripe（決済/サブスクリプションの管理）
- メール送信: SMTP（開発はMailhog想定）
- 地図: Google Maps APIキー（必要時）
- キャッシュ/集計: Redis（必要時）
- 画像/ファイル: S3またはローカルストレージ（設定で切替）

#### 4.3 制約（運用・安全）
- パスワードはハッシュ化して保存（bcrypt）
- アップロードファイルは最大10MB、許可拡張子を限定

### 5. 機能要件（ユーザー/店舗/運営）
#### 5.1 認証・アカウント
- 会員登録・ログイン
- SNS連携ログイン（一般ユーザーのみ）
- パスワード再設定・メール認証（トークン管理）
- Access/Refreshによるセッション維持

#### 5.2 ユーザー向け機能
- 地図/検索/絞り込み（市区町村・特徴・色味などの条件）
- 店舗詳細閲覧（営業時間、料金、設備、混雑/空席、写真等）
- お気に入り（店舗リストの作成・編集・複数リスト管理）
- 閲覧履歴
- 求人掲示板と求人のお気に入り
- 通知（プラットフォーム内通知を想定）
- 決済管理（プラン切替・支払い履歴・支払い方法）
- 通報/違反報告
- 問い合わせ（ユーザーから運営または店舗へ）

#### 5.3 店舗向け機能
- 店舗掲載依頼（アプリケーション提出）
- 掲載申請のステータス管理（運営承認/却下/保留）
- 店舗情報更新（住所/連絡先/営業時間/画像等）
- 空席状況のワンクリック更新（一般ユーザーに表示）
- お知らせ/日記の投稿（下書き・予約・編集）
- セール情報の投稿（テンプレート保存・再利用）
- 自店舗インサイト（ダッシュボード）
- 決済管理（店舗向け支払い・領収書PDF発行）
- 問い合わせ対応（運営へ）

#### 5.4 運営向け機能（管理画面/API）
- ダッシュボード統計（ユーザー/店舗/売上/通報件数）
- ユーザー管理（有効化/停止/ステータス更新）
- 店舗管理（掲載申請の承認/反映）
- 通報管理（通報の一覧/ステータス更新/処理）
- 決済・レポート関連の参照（必要に応じて）
- 権限・アクセス制御（管理者のみ）

### 6. 非機能要件
#### 6.1 セキュリティ
- JWT（Access/Refresh）とトークン失効管理（Refresh token revoke）
- RBAC に基づくAPI保護
- 管理系操作は管理画面/API経由に限定
- パスワードは `bcrypt` でハッシュ化

#### 6.2 性能/スケーラビリティ
- 一覧系APIはページング（`page` / `page_size`）
- ランキングは毎日バッチ計算（リアルタイム計算回避）
- イベントログは将来の集計/分析を前提に保存

#### 6.3 可用性・保守性
- モジュラーモノリス構成（ドメインモジュール分割）
- OpenAPI駆動によるフロント型同期（endpointズレの抑制）
- 移行はAlembicで管理

### 8. KPI/受入基準（既存要件からの要約）
本システムは KPIラダー/NSM を用いて、店舗側の継続（混雑投稿/分析閲覧）と、ユーザー側の来店意図行動（店舗詳細遷移）を中心に評価する。

#### 8.1 最終目標（5年目）
ユーザー登録者数（推奨/保守/挑戦）を以下とする（「東京都在住・通勤通学圏内」を主対象、関東近県/訪日客で上振れ余地あり）。

- Base（推奨目標値）: 17万人
- Conservative: 11万人
- Stretch: 26万人

#### 8.2 KPIラダー（年次マイルストーン）
ユーザー獲得・登録化の年次マイルストーンを以下とする。

- 1年目: 2.5万人（α/β + 都心5区で集中露出）
- 2年目: 5.5万人（SEO強化 + 投稿UGC施策）
- 3年目: 9.0万人（店舗ポータル普及 + 混雑状況の定常化）
- 4年目: 13万人（関東近県 + 訪日導線）
- 5年目: 17万人（機能拡張: レビュー軽量化 + プッシュ最適化）

#### 8.3 店舗KPI（都内約6,121店舗を分母にファネル設計）
都内カフェ（約6,121店舗）を分母として、獲得→活性化→継続/貢献→価値（送客）→品質/運用の5層で設計する。

1) 獲得
- Verified Rate（本人確認完了率）: 本人確認完了店舗 / 登録店舗数
- Onboarding Completion（初期設定完了率）: プロフィール80%以上・営業時間・写真2枚以上 を満たす店舗の割合

2) 活性化
- TTFU（Time To First Update）: 登録申請完了 → 初回「混雑状況投稿」までの中央値（時間）
- W1 Post Rate: 登録申請完了後7日以内に混雑状況投稿 >= 1 の割合

3) 継続/貢献
- WAO（Weekly Active Owners）: 週1回以上、混雑投稿 or 分析閲覧を実施したオーナー数 / 率
- Post Freq.（混雑投稿頻度）: アクティブ店舗の週次中央値（件数/店舗数/週）
- 営業更新率: 営業日で1日1回以上混雑更新があった日の割合（店舗単位→全体平均）
- Analytics MAO率: 月1回以上ダッシュボード閲覧したオーナー率

4) 価値（送客）
- Route Clicks/Active Store/Week: アクティブ店舗あたり週の経路クリック
- Owner-Attributed Lift: 「混雑更新後6時間以内」の経路クリック上昇分（差分寄与）
  - 算出式: 実測 - 直前7か同時間帯平均
- Favorites Gain: アクティブ店舗の週次「お気に入り」純増率/数

5) 品質運用
- Fresh Coverage（鮮度カバレッジ）: 営業時間内の「最新更新が60分以内」の時間割合（店舗谷→全体平均）
- Moderation MTTR: 通報 → 一次対応の平均時間
- Churn30: 30日間「ログイン&投稿なし」の休眠化率（登録申請済み総数比）

#### 8.4 中間KPI/NSM（受入観点）
KPI/NSMは「測定可能であること」「ダッシュボード/管理画面で追跡可能であること」を受入観点とする。

- KPI（例）
  - CVR（メイン画面 → 店舗詳細）
  - CVR（地図画面 → 店舗詳細）
  - 日次/週次/月次 混雑状況更新店舗数
  - 混雑データ鮮度（中央値）
  - 店舗オーナー継続率
  - ユーザー増減率 / 登録者数推移
- NSM
  - 地図からの「来店意図アクション」
    - メイン画面 → 店舗詳細
    - 地図画面 → 店舗詳細
