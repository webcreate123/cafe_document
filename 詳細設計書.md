## 詳細設計書（Cafe Platform）

作成日: 2026-03-25

### 0. 本書の目的
本書は、`cafe_web / cafe_admin` の画面操作と `cafe_api` のAPI、DB（`cafe_api/app/modules/**/models.py`）を対応付け、実装・運用・保守で必要となる詳細仕様を整理する。

本ドキュメントは「画面/API/DB」を横断して説明するため、画面単位の手順に加え、APIエンドポイント単位の入出力・権限・例外を明記する。

---

### 1. 共通仕様（認証・認可・リクエスト/レスポンス・ログ）
#### 1.1 API基準（prefix）
`cafe_api` は統合ルータにより、以下prefixでグループ化する（実際のURLは `app/api/v1/api.py` により `/api/v1` が付与される）。

- `/auth`
- `/users`
- `/stores`
- `/favorites`
- `/jobs`
- `/sales`
- `/analytics`
- `/payments`
- `/notifications`
- `/reports`
- `/admin`
- `/history`
- `/inquiries`
- `/announcements`

#### 1.2 認証（JWT）
`cafe_api/app/core/dependencies.py` で `HTTPBearer` により `Authorization: Bearer <access_token>` を受け、token payload の `type` が `access` であることを検証する。
不正/期限切れ/ユーザー不存在の場合は `HTTP 401` を返す。

`cafe_web/src/lib/api-client.ts` では、`access_token` をCookie `access_token` に格納し、401が発生した場合に `/auth/refresh` を叩いて自動更新する（同時リフレッシュはPromise再利用で抑制）。

#### 1.3 認可（RBAC）
許可ロールは `cafe_api/app/core/constants.py` の `UserRole` に従う。

- `USER`
- `STORE_OWNER`
- `STORE_STAFF`
- `ADMIN`

エンドポイントの制御は `require_role()` を利用し、`/admin`配下・管理操作に `ADMIN` 限定を適用する。
ユーザーが非アクティブの場合は `get_current_active_user` により `HTTP 403` を返す。

#### 1.4 エラー設計
各routerは原則として以下のパターンを取る。

- 見つからない: `HTTP 404` + `ErrorMessage` 定数
- 権限不足: `HTTP 403` + `ErrorMessage.INSUFFICIENT_PERMISSIONS` 等
- バリデーション/処理失敗: `HTTP 400` + `detail=str(e)`

### 3. 詳細設計：認証・アカウント（一般ユーザー）
#### 3.1 会員登録（POST /auth/register）
権限: 誰でも可

入力: `RegisterRequest`
処理:
- `AuthService.register()` が `users` へユーザー作成
出力: `{"message": "...", "user_id": ...}`

例外:
- メール重複等は `HTTP 400`

#### 3.2 ログイン（POST /auth/login）
権限: 誰でも可

入力: `LoginRequest`
処理:
- `AuthService.login()` が認証し、Access/Refresh token を返却
出力: `Token`（access_token / refresh_token）

例外:
- 認証失敗は `HTTP 401`

#### 3.3 トークン更新（POST /auth/refresh）
入力: `RefreshTokenRequest`
処理:
- Refreshトークン検証後、Access token発行
出力: `Token`

#### 3.4 ログアウト（POST /auth/logout）
入力: `RefreshTokenRequest`
処理:
- Refresh token を無効化（`refresh_tokens.is_revoked` 更新）
出力: `HTTP 204`

#### 3.5 パスワードリセット
- 申請（POST /auth/password-reset/request）: メール送信（開発はMailhog想定）
- 確認（POST /auth/password-reset/confirm）: トークン検証→パスワード更新

### 4. 詳細設計：店舗（オーナー/スタッフ）
#### 4.1 店舗掲載申請（POST /stores/register）
権限: 誰でも可（未ログイン可想定の public）

入力: `StoreRegistrationRequest`
処理:
- `StoreService.request_registration()` により `stores.registration_status` を `pending` として作成/更新
出力: `StoreRegistrationResponse`（`id / name / registration_status / registration_requested_at / message`）

例外:
- バリデーション失敗等 `HTTP 400`

#### 4.2 自分が管理する店舗の取得（GET /stores/me/stores）
権限: 認証必須（`get_current_active_user`）

処理:
- `StoreService.list_managed_stores()`（所有店舗 + 資格情報で追加した店舗）
出力:
- `StoreListResponse`（`stores / total / page / page_size`）

#### 4.3 店舗一覧・地理検索（GET /stores）
権限: 任意（ユーザー有無で `user_id` を渡してお気に入り等の文脈を反映する方針）

クエリ:
- `page` / `page_size`
- `latitude` / `longitude` / `radius_km`（指定時のみ半径検索）

出力:
- `StoreListResponse`（TODO: `total` は暫定 `len(stores)`）

#### 4.4 店舗詳細閲覧（GET /stores/{store_id}）
権限: 任意（`get_current_user_optional`）

処理:
- `StoreService.get_store_with_view_recording(store_id, user_id)` が閲覧履歴/分析イベントを記録しつつ店舗を返す
例外:
- 存在しない場合 `HTTP 404` + `ErrorMessage.STORE_NOT_FOUND`

#### 4.5 店舗情報更新（PUT /stores/{store_id}）
権限:
- `STORE_OWNER` / `STORE_STAFF` / `ADMIN`（`require_role` ではなくエンドポイント内で can_manage を判定）

処理:
- `StoreService.can_user_manage_store(current_user.id, store_id)` で管理可否判定
- 更新は `StoreService.update_store()`

例外:
- 管理権限なし: `HTTP 403` + `ErrorMessage.NOT_AUTHORIZED_TO_UPDATE_STORE`
- 存在しない: `HTTP 404`

更新対象のDB（storesテーブル）:
- `address / phone / website / opening_hours / images / prefecture / city / is_active / management_password_hash` 等

#### 4.6 空席状況のワンクリック更新（POST /stores/{store_id}/vacancy）
権限:
- `STORE_OWNER` / `STORE_STAFF` / `ADMIN`

処理:
- `StoreService.toggle_vacancy(store, current_user.id)`
- `stores.is_vacant` をトグル

出力:
- `StoreResponse`（最新の店舗情報）

#### 4.7 店舗のトップ表示（GET /stores/top/*）
目的: `cafe_web` トップ画面のカード群（新着/ランキング/セール/featured）

エンドポイント:
- `/stores/top/new`
- `/stores/top/ranking`
- `/stores/top/sales`（クエリ: `prefecture`, `city`）
- `/stores/top/featured`

補足:
- featured は `stores.is_featured`（運営が切替）

#### 4.8 運営選定切替（POST /stores/admin/{store_id}/featured）
権限: `ADMIN`
処理:
- `StoreService.toggle_featured_status(store, is_featured)`

### 15. 非同期・バッチ（workers）の詳細設計
#### 15.1 アナウンス公開ワーカー（announcement_worker）
実装:
- `run_announcement_worker()` が1分間隔で `publish_scheduled_announcements()` を実行
- `announcements.scheduled_at` が到達したものを `published_at` 更新などで公開

#### 15.2 ランキング更新ワーカー（ranking_worker）
実装:
- 24時間間隔で `Store.is_active=true` の店舗を対象にランキングを再計算
- スコア算出: `favorite_count*0.4 + route_click_count*0.3 + view_count*0.3`
- `store_rankings`（`score / rank / updated_at`）と `stores.ranking_score / ranking_updated_at` を更新

#### 15.3 クリーンアップ（cleanup_worker）
実装:
- Refresh tokenの期限切れ削除（`refresh_tokens.expires_at < now`）
- イベントログの古いデータ削除（デフォルト90日保持）

#### 15.4 メールワーカー（email_worker）
現状:
- `arq` を利用する `python -m app.workers.email_worker` エントリが存在する
- `entrypoint.sh` は `SERVICE=email-worker` または `SERVICE=both` で起動可能
