# 基本設計書（Cafe Platform）

## 1. システム概要
- 本システムは、`cafe_web`（利用者/店舗向けWeb）、`cafe_admin`（運営管理画面）、`cafe_api`（FastAPI）で構成されるカフェプラットフォームである。
- 主機能は、店舗検索・閲覧、店舗登録申請、セール/お知らせ投稿、求人管理、お気に入り、閲覧履歴、問い合わせ、通知、決済、管理者運用である。
- APIは`/api/v1`配下に機能別ルーターを持ち、`cafe_web`と`cafe_admin`はOpenAPI生成クライアントを通じて利用する。

## 2. システム構成

### 2.1 フロントエンド
- `cafe_web`: Next.js App Router + localeルーティング（`app/[locale]`）
  - 一般ユーザー向け: マップ、店舗詳細、お気に入り、閲覧履歴、求人閲覧
  - 店舗向け: ダッシュボード、店舗設定、セール、お知らせ、求人、問い合わせ、決済
- `cafe_admin`: Next.js App Router
  - 管理者向け: ダッシュボード、ユーザー管理、店舗承認、通報管理、決済/分析

### 2.2 バックエンド
- `cafe_api`: FastAPI
  - ルーター: auth/users/stores/favorites/jobs/sales/analytics/payments/notifications/reports/admin/history/inquiries/announcements
  - ORM: SQLAlchemy
  - マイグレーション: Alembic
  - 非同期ワーカー: announcement, analytics, ranking, cleanup, email

### 2.3 外部連携
- 決済: Stripe（PaymentIntent/Subscription/Webhook）
- メール: SMTP（開発時はMailhog利用可能）
- 画像/ファイル: ローカルuploadsまたはS3（設定依存）
- OAuth: ソーシャルログイン（OAuth/Apple）

## 3. 利用者区分と権限
- 一般ユーザー（`USER`）: 店舗閲覧、履歴、いいね、求人閲覧、問い合わせ等
- 店舗オーナー/スタッフ（`STORE_OWNER`,`STORE_STAFF`）: 店舗情報更新、セール/お知らせ/求人運用
- 管理者（`ADMIN`）: ユーザー状態変更、店舗承認/拒否、通報対応、統計閲覧

## 4. 機能一覧
- 認証機能: 会員登録、ログイン、トークン更新、ログアウト、メール認証、パスワードリセット、SNS認証
- 店舗機能: 店舗CRUD、近傍検索、トップ表示（新着/ランキング/セール/特集）、空席切替、店舗申請/承認
- お気に入り機能: リスト管理、店舗お気に入り登録/解除、一覧取得
- 求人機能: 求人CRUD、店舗別/お気に入り求人取得、求人お気に入り
- セール機能: セールCRUD、画像アップロード、テンプレート管理
- お知らせ機能: 投稿CRUD、下書き、予約投稿、カテゴリ管理
- 履歴機能: 店舗閲覧履歴参照/削除
- 問い合わせ機能: 問い合わせ登録/参照/更新
- 通知機能: 通知取得、未読数、既読化
- 決済機能: PaymentIntent、Webhook、決済履歴、サブスク管理
- 管理機能: ダッシュボード統計、ユーザー管理、店舗一覧、通報管理
- 分析機能: ユーザー/店舗イベント記録、店舗ダッシュボード、全体統計

## 5. 非機能要件（実装反映）
- セキュリティ
  - Logging/SecurityHeadersミドルウェア
  - CORS設定、TrustedHost設定
  - 認証トークンベース（access + refresh）
- 可用性/運用
  - `/health`ヘルスチェック提供
  - バッチ/ワーカーでランキング・告知配信等を分離
- 拡張性
  - 機能ごとのモジュール構造
  - OpenAPIコード生成によりフロントとAPI仕様を同期

## 6. データ設計方針
- 全テーブルで`id`,`created_at`,`updated_at`を共通保持（`BaseModel`）。
- 論理的なドメイン単位（user/store/favorite/job/sale等）にテーブルを分割。
- 管理系・履歴系・監査系（events/reports/history/notifications）を独立させ、分析や運用要件へ対応。

## 7. API設計方針
- RESTベースで機能単位のプレフィックスを採用（例: `/api/v1/stores`）。
- 一覧系は`page`,`page_size`を基本としたページネーション。
- 役割が明確な拡張エンドポイントを採用（例: `/stores/{id}/vacancy`,`/stores/admin/{id}/approve`）。
