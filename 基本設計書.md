## 基本設計書（Cafe Platform）

作成日: 2026-03-25

### 1. 全体アーキテクチャ
本システムはモジュラーモノリス型のバックエンド（`cafe_api`）と、Next.js（`cafe_web` / `cafe_admin`）によるフロント分離で構成する。

- バックエンド: FastAPI + SQLAlchemy + Alembic
- フロント: Next.js（App Router）+ OpenAPI自動生成クライアント
- DB: PostgreSQL + PostGIS（店舗位置情報検索）
- 分析/集計: イベントログ保存（`user_events`, `store_events`）+ バッチ（ランキング等）

#### 1.1 ディレクトリ/ドメイン設計（バックエンド）
`cafe_api` はドメインモジュール単位で分割する。

- `auth`: 認証・トークン管理
- `user`: ユーザー情報
- `store`: 店舗・ランキング・空席フラグ等
- `favorite`: お気に入りリスト/お気に入り
- `history`: 閲覧履歴
- `sale`: セール
- `job`: 求人
- `inquiry`: 問い合わせ
- `announcement`: お知らせ/日記
- `notification`: 通知
- `payment`: 決済・サブスクリプション
- `analytics`: イベントログ基盤
- `report`: 通報
- `admin`: 運営管理API

### 2. データフロー（主要ケース）
#### 2.1 ログイン/認証
1. フロントが `cafe_api` の `/auth` を呼び出す
2. Access token（短命）と Refresh token（長命）を発行
3. フロントはJWTを保持し、以降のAPI呼び出しで認証する

#### 2.2 店舗検索〜詳細閲覧
1. ユーザーが地図/検索条件入力（`cafe_web`）
2. フロントが `cafe_api` の `/stores` または関連APIを呼び出す
3. 店舗詳細を取得し、いいね/お気に入り/通報などを実行
4. 閲覧履歴/イベントログを保存し、分析に利用する

#### 2.3 店舗の投稿（お知らせ/セール/空席）
1. 店舗がログイン後、該当ページから投稿
2. `cafe_api` に `/announcements` または `/sales` または店舗更新系APIを呼び出し
3. DB更新と同時にイベントログを保存する（必要時）
4. 一般ユーザーは地図/店舗詳細で最新情報を参照する

### 3. 認証・認可設計（RBAC）
#### 3.1 ロール
`UserRole` に基づき以下を扱う。

- `USER`, `STORE_OWNER`, `STORE_STAFF`, `ADMIN`

#### 3.2 実装方針
- API層で `require_role` 依存性注入によりアクセス制御する
- 管理者系API（`/admin`）は `ADMIN` 限定
- 店舗関連の操作は `STORE_OWNER` / `STORE_STAFF` の権限に応じて許可

### 4. バックエンド設計詳細
#### 4.1 API設計方針
- ルーティングは prefix でドメイン分割（`/stores`, `/favorites`, `/payments` 等）
- Pydantic schema によりリクエスト/レスポンスを明確化
- 例外はHTTPExceptionに変換し、エラーコード/メッセージを返す

#### 4.2 データアクセス方針
- 各モジュールに repository / service / router を持つ（責務分離）
- 一覧取得は offset/limit（ページング）
- ランキングや統計はバッチ/集計クエリで計算（リアルタイム計算回避）

### 5. フロントエンド設計詳細
#### 5.1 OpenAPI自動生成クライアント
`cafe_api` の `openapi.json` から以下を自動生成する。

- TypeScript型
- APIクライアント関数

フロントは生成物のみを利用し、手書き型やendpointズレを防ぐ。

#### 5.2 React Query等（利用前提）
既存構成ではReact Queryと組み合わせる運用を想定する（`src/hooks` 側に実装がある）。

### 6. マスタ/コード値の管理
列挙値（ロール、ステータス、通知種別など）は `cafe_api/app/core/constants.py` と各モデルの Enum 定義に従う。

### 7. 運用・保守性設計
- DBマイグレーション: Alembic
- バッチ: `cafe_api/app/workers`（ランキング等）
- イベントログ: 溜めて分析する前提でスキーマ拡張容易性を確保
